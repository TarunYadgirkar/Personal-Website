### C. Visual style & animation

**Overall feel:** engineered, precise, quietly confident. "Lab notebook meets modern product site." Aim closer to Caglar/Chiang/Feng restraint than Freiberg spectacle.

**Palette (pick one direction):**
- *Technical dark* — near-black/deep-navy base, one restrained accent (electric blue, or an amber/green "terminal" accent that nods to embedded/hardware). Reads engineered.
- *Editorial light* — warm off-white, near-black text, single saturated accent. Reads researcher/founder, more VC-legible.
- Either works; **commit fully and keep the accent to one color.**

**Type:** a clean grotesque for body (Inter / Geist / Söhne-like) **plus a monospace** for metadata, tags, section labels, and numbers. The mono is your cheap "systems" signal (echoes Caglar's ASCII instinct) — use it for tech tags, dates, status labels, publication venues. Optionally one serif for large display headers if you go editorial-light. Big type scale, generous line-height, lots of whitespace.

**Signature element — choose exactly one, and only if it maps to real work:**
- A subtle **Spline / react-three-fiber 3D accent** in the hero — but make it *mean* something: a slowly rotating point cloud, a stylized PCB/robot mechanism, or an abstract sensor-mesh. **Not** a generic floating blob. Keep it low-motion, muted, non-blocking; ship a static fallback and respect `prefers-reduced-motion`.
- **Or** a text/monospace signature (an ASCII/generative masthead à la Caglar) — cheaper, faster, and arguably more distinctive for your audience than yet another 3D hero.
- Decision rule: 3D is worth it *only* if it visualizes something you actually built. If it's ornament, skip it — ornament reads as compensating.

**Motion (adopt Kowalski's rules):** entrances ease-out, <300ms; springs for anything physical; ~0.97 scale-down on press; a restrained hover elevation on cards; smooth section transitions. Motion should feel *robust* (survives fast scroll / spam clicks) — untested jank actively lowers credibility. **Accessibility:** honor `prefers-reduced-motion`.

**Anti-slop checklist (do NOT do these):**
- Pulsing/glowing/breathing status dots or "AI active" indicators
- Blur-everywhere or fade-up-everything entrances
- `ease-in` on enter animations
- Animating high-frequency actions
- Auto-playing sound; parallax for its own sake; typewriter effects on the hero
- Decorative 3D blobs, particle fields, gradient meshes with no meaning
- Badge walls / logo soup of technologies
- Emoji as UI; "🚀 passionate about 🤖 robotics" energy

### E. Project card structure

Standard card (list/grid), designed for fast scanning by non-experts *and* depth for technical readers:

```
[ thumbnail / looping muted video / system diagram ]
Precise Technical Title                         [status tag]
One-line: what it is + why it matters (≤ ~18 words)
Role · context (lab / hackathon / competition / independent)
[ tech tags — monospace ]  [ outcome tag: Patent · Paper @venue · VEX Worlds place · Award · N users ]
→ links: [Paper] [Patent] [GitHub] [Demo] [Devpost] [Video]
```

Rules:
- **Video-first for anything physical.** A 30–60s muted, looping clip of the robot/hardware working beats any still (core robotics-portfolio guidance). Include a system/architecture diagram where relevant.
- **Status tags** (Kalakuntla): `Patent granted` · `Published` · `Shipped` · `In progress` · `Prototype`.
- **Outcome tag carries the credibility** (Kowalski): the patent/paper/placement/number, not adjectives.
- **Precise names** (Kalakuntla): "Assistive Balance-Rehabilitation Exoskeleton," not "Cool Health Robot."
- Flagship cards on the homepage get a full detail page (`/projects/[slug]`): problem → approach → system → results/outcome → media → links. Archive items can be one line + link.

### I. Recommended tech stack

**Primary recommendation (best ceiling, industry-legible):**
